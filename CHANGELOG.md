## [1.4.0] - 2026-03-24

### 🏠 本地 LPCC 编译
- 新增 `本地LPCC编译当前文件` 命令与按钮，用于在 VS Code 内直接对当前文件执行离线 LPCC 编译。
- 本地编译只围绕当前文件所属的 mudlib 根目录工作：自动扫描 `lpcc.exe`、`config.ini`、`config.cfg`，手动选择和保存的路径也限定在当前 mudlib 内，不再去管其他工程目录。
- 新增 `本地LPCC设置` 入口，把 `LPCC 路径`、`配置文件路径`、`保存自动本地编译`、`是否提示警告`、`诊断提示语言` 合并到一个按钮里统一管理，并把结果保存到当前项目的 VS Code 工作区设置。
- 新增 `gameServerCompiler.localCompile.autoCompileOnSave`，默认关闭；保存 `.c/.lpc` 时会静默执行本地 LPCC 编译，仅在 LPCC 和配置文件已明确可用时触发，缺失、无效或多候选未确认时自动跳过，不会弹窗打断保存。
- 手动“保存后编译”场景会自动跳过本次保存触发的本地自动编译，避免一次点击造成两次本地编译。

### 🌏 编译诊断中文化
- 新增 `gameServerCompiler.diagnostics.messageLanguage` 设置，默认 `dual`，支持切换为 `en` 或 `zh`。
- 本地 LPCC、远程编译消息、Problems 与输出摘要现在共用同一套驱动诊断翻译规则。
- 新增 `COMPILER_DIAGNOSTIC_LOCALIZATION.md`，集中维护驱动英文原文、插件中文提示与来源文件的对照表。

### ✨ 函数声明工作流
- 新增 `生成当前文件函数声明` 命令，并在控制台把按钮放到 `复制相对路径` 旁边，改为按需手动刷新声明块。
- `gameServerCompiler.compile.autoDeclareFunctionsOnSave` 继续保留，但默认值改为关闭；需要时可在 VS Code 设置中重新开启保存时自动声明。
- 重新生成声明块时，会把文件里与当前函数定义匹配的散落声明一并收拢进 `AUTO DECLARATIONS` 块，避免声明散落在正文各处。
- 收紧函数头匹配规则，避免把上一行脏文本误吞进声明签名，修复 `aaaavoid int main();` 这类错误声明。

### 🧭 输出、面板与交互
- 常规输出统一收敛到单一 `LPC-MUD工具` 输出栏，本地与远程编译共用同一输出通道；编译摘要格式统一，非编译操作只输出结果，不再刷标题噪音。
- 指令控制台按“本地命令 / 远程命令”重排，配置区改为点击展开，并补充显示当前 LPCC、当前 Config、保存自动本地编译、警告提示与诊断语言。
- `编译当前文件` 命令标题调整为 `远程Update当前文件`，避免与本地 LPCC 编译混淆。
- 文件相关命令不再严格依赖编辑器焦点：当焦点切到输出栏、Problems 或侧栏时，仍会优先使用当前可见的代码文件执行远程 update、本地编译、复制相对路径和生成函数声明。

### ✅ 验证
- `npm run compile` 通过
- `npm run lint` 通过

## [1.3.3] - 2026-03-24

### 🧭 编译诊断与定位
- 编译错误解析链路升级：优先识别 FluffOS 原始诊断头 `/file line N[, column M]: [Warning: ]message`，直接提取文件、行号、列号与错误消息。
- 保留对现有 mudlib 中文编译块的兼容回退：即使服务器最终返回的是 `编译位置 / 编译原因 / 源码片段 / 触发链` 这种格式，插件也会压缩为一条准确诊断，不再把整块噪音逐条刷进消息面板。
- 错误定位统一改为行列级别定位；若编译器返回列号，则点击卡片和 Problems 都会直接跳到具体列。

### 🎯 消息面板与 Problems
- 编译失败时，消息面板只保留一条可点击摘要，例如 `❌ /cmds/wiz/testcmd.c:584:16 syntax error ...`。
- 新增 VS Code 配置项 `gameServerCompiler.ui.autoRevealProblems`：
  - `never`：不自动弹出
  - `error`：仅编译错误时弹出
  - `errorOrWarning`：错误或警告都弹出
- 恢复简洁的“正在编译: /path”提示，同时移除多余的编译过程噪音消息，避免侧边栏被状态块淹没。

### 🛣️ 路径与项目根目录
- 修复“自动识别根目录成功，但点击错误卡片打开文件仍按工作区根目录拼路径”的问题。
- 诊断卡片点击跳转、Problems 诊断写入、本地文件打开三条链路统一复用同一份本地绝对路径，兼容 `mud_nextB/duobao` 这类工作区根目录与真实 MUD 根目录不一致的场景。
- 修复 Windows 下 `toLocalPath()` 处理 `/cmds/...` 这类 mud 路径时的本地路径解析错误。

### ✅ 验证
- `npm run compile` 通过
- `npm run lint` 通过
- 补充并验证编译诊断、Problems 自动弹出策略、mudlib 中文编译块兼容回退、MUD 本地路径解析的测试与手工断言

## [1.3.2] - 2026-03-20

### ✨ 新增功能
- 保存 `mudlib` 范围内的 `.c` 文件时，自动扫描当前文件中的函数定义，并在文件顶部生成或刷新 `AUTO DECLARATIONS` 声明块。
- 支持提取常见 LPC/C 风格函数定义，包括 `protected int foo(...)` 与无返回类型的 `create()` 这类写法。

### ⚙️ 保存与编译
- 声明生成逻辑接入保存前阶段：写盘前先更新声明块，避免“保存一次改代码、再保存一次才生效”的割裂体验。
- 保存后的自动编译链路保持不变；若已开启 `autoCompileOnSave`，声明更新完成后仍会继续走原有编译流程。
- 新增配置项 `gameServerCompiler.compile.autoDeclareFunctionsOnSave`，默认开启，可按需单独关闭自动声明生成。

### 🧹 兼容性与细节
- 仅对 `.c` 文件生效，避免影响 `.lpc`、配置文件和普通文档。
- `mudlib` 判断兼容两种场景：路径中显式包含 `mudlib`，或当前文件能被现有 MUD 项目根自动识别逻辑识别出来。
- 已存在的自动声明块会被原位刷新；若文件内已无函数定义，则会自动移除旧声明块。

### ✅ 验证
- `npm run compile` 通过
- `npm run lint` 通过
- `npm test` 通过（19/19）

## [1.2.0] - 2026-01-28

**🎨 用户体验改进版本**
本次更新专注于用户体验和配置管理的全面提升，解决了多个反馈问题。

### ✨ 新增功能

#### 配置管理优化
- **延迟创建配置**：不再于打开项目时自动创建，仅在首次连接服务器时生成，减少干扰。
- **重置项目路径**：
  - 新增界面按钮："重置项目路径"。
  - 新增 VS Code 命令：`LPC服务器: 重置项目路径`。
  - 方便项目文件夹移动后快速修正配置。
- **配置信息面板**：
  - 在"连接服务器"下方直观显示当前配置。
  - 包含：项目路径（工作目录）与 连接地址（IP:Port）。
  - 采用高亮配色与等宽字体，提升可读性。

#### 配置实时更新
- **初始化监听**：插件启动时即挂载监听器，配置更改立即生效，无需重启。
- **全量监听**：支持监听 `rootPath`, `host`, `port`, `encoding` 等所有字段变化。
- **防抖机制**：添加 200ms 防抖及文件大小检查，确保读写稳定性。

### ⚡ 改进与优化

#### 文件保存与编译
- **智能文件检查**：优先检查文件后缀，非 `.c`/`.lpc` 文件直接跳过，避免配置文件触发错误的编译日志。
- **静默处理**：优化保存流程，仅在真正的读取失败时输出错误，忽略常规的中间状态。

#### 消息与显示
- **去重提示**：修复了 `addMessage` 重复调用的问题，连接失败等关键信息只显示一次。
- **ANSI 清理增强**：
  - 优先使用通用正则清理所有 ANSI 转义序列。
  - 彻底移除残留的颜色代码（如 `[1m` 等），确保日志纯净。

### 🐛 问题修复
- 修复修改配置文件后不生效的问题（引入 `isWatchingConfig` 标志）。
- 修复保存配置文件时偶发的 "配置文件为空" 误报。
- 修复非编译文件触发编译检查日志的问题。
- 修复连接地址偶尔显示为 "未配置" 的 UI 用于问题。

### 🔧 技术细节
- **ConfigManager.ts**：构造函数内启动监听，增强变化检测，增加详细调试日志。
- **extension.ts**：优化文件监听顺序，`convertToMudPath` 强制获取最新配置。
- **tcpClient.ts**：重构 `cleanColorCodes` 逻辑，增加残留清理正则 `\[[0-9;]*[mK]`。
- **buttonProvider.ts**：UI 组件增加配置监听，实现界面自适应更新。
